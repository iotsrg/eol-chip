What is EOL-CHIP?

EOL-CHIP is a free, public catalog of end-of-life (EOL) and obsolete chips - microprocessors, microcontrollers, FPGAs, network controllers, sensors, radios, and SoCs. For each chip, we link out to the security advisories, CVEs, public exploits and vendor disclosures that target it.

Think of it as a Wikipedia for legacy silicon, with a security lens. Every chip has its own article page with:

  • Manufacturer, category, lifecycle status (Active / Last Buy / EOL Announced / Obsolete)
  • Wikipedia summary + Wikidata facts (release year, process node, transistor count, predecessor / successor)
  • Linked vulnerabilities (NVD CVEs, CISA KEV, Exploit-DB, Metasploit, GHSA, ICS, Packet Storm, OSV)
  • External links to datasheets, manufacturer pages, and FCC ID lookups

Why does this matter?

When a chip goes end-of-life, manufacturers stop shipping firmware updates. The chip often lives on in the wild for another 10–20 years inside routers, medical devices, industrial controllers, automotive ECUs, smart-home hubs, drones, and game consoles. Once a vulnerability is published, those devices stay vulnerable forever.

EOL-CHIP makes it easy to look up any obsolete part by name and immediately see the active threats against it - without digging through ten different vulnerability databases.

How it works

Every day, a GitHub Action fetches fresh data from public security feeds and enrichment sources. The pipeline runs in this order:

  1. Fetch - 10 separate scripts pull CVEs, exploits, KEV listings, Metasploit modules, GHSA advisories, ICS-CERT bulletins, Packet Storm, OSV, vendor PSIRTs (Cisco, Fortinet, etc.), and NHTSA vehicle recalls.
  2. Process EOL - convert the human-maintained chips.yaml file into the structured chip catalog.
  3. Enrich - for every chip, query Wikipedia REST + Wikidata for a summary, photo, and structured specs (process node, clock frequency, predecessor, etc.).
  4. Cross-link - match chip part numbers against every threat record. A chip ends up tagged with how many CVEs / KEV listings / exploits / Metasploit modules reference it.
  5. Index & deploy - build a single search index and publish to GitHub Pages.

The site itself is plain static HTML/CSS/JS - no backend, no tracking, no cookies. All filtering and search runs in your browser.

How often it updates

The pipeline runs daily at 06:00 UTC via GitHub Actions. Every fetcher is fault-tolerant: if a source returns nothing, we keep the last known good data (no wiping with empty arrays). If a critical source returns 0 results, the workflow auto-opens a GitHub issue tagged data-fetch-alert so we know to investigate.

You can see the freshest "Last updated" timestamp in the footer of every page.

How to use it

  • Look up a chip - type its part number in the homepage search box (e.g. ESP32, i9-13900K, Z80).
  • Browse by manufacturer - open the Vendors page, click any vendor (Intel / Microchip / NXP / Espressif / …), then click any chip card to open its article.
  • Read a chip article - every chip has a wiki-style page: intro, photo, infobox with structured specs, lifecycle, security section listing all known CVEs / KEV / exploits / Metasploit modules, and external links to datasheets.
  • Search across everything - homepage search covers all 4,000+ entries (CVEs, exploits, chips, vendor advisories). Use the filter pills to narrow by source.

How to contribute

The project is open source. The entire codebase, data files, and chip catalog live at github.com/iotsrg/eol-chip.

  • Add a chip - edit chips.yaml and submit a pull request. One YAML block per chip with part_number, name, manufacturer, category, description, status, and (where known) eol_date + last_order_date.
  • Add a data source - drop a new scripts/fetch_*.py following the same pattern, register it in build_search_index.py and the GitHub Action.
  • Fix data - wrong EOL date, missing chip, broken link? Open an issue or PR.

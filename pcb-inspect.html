How accurate is this?

⚠ Image quality matters.
  • Good: top-down, sharp, even light, silkscreen readable.
  • Bad: angled, blurry, glare, low-res.
  • Schematics: accuracy drops further.
  • Small models hallucinate more — use Opus / Gemini 2.5 Pro / 90B Ollama.
  • Part identification is good when silkscreen is sharp and the package is common (QFN / QFP / BGA / SOIC from major vendors). Custom-marked or sanded chips will fail. Date codes and lot numbers are often misread.
  • Bounding boxes are approximate — vision models are not pixel-perfect localisers. Use them as visual pointers, not measurements. Toggle edit on the image to drag/resize a box manually if you need it tight.
  • Pin labels are inferred from package + reference design, not literally traced from the photo. Always confirm with the datasheet before probing.
  • Debug interfaces (UART / JTAG / SWD / ISP / TagConnect) are identified visually: by pin count, silkscreen labels (TX / RX / SWDIO / SWCLK / RST), proximity to the MCU, and population state. We can NOT electrically trace tracks from a photo — that needs a multimeter. The model will say “likely UART” or “consistent with SWD”, never assert wiring it can't see. Verify pinout with a continuity probe before connecting anything.
  • I²C buses are usually inferred from clues rather than seen directly — SDA/SCL labels on test points, I²C-family chips visible on the board (24Cxx EEPROM, DS3231 RTC, BMP/BME/SHT/MPU sensors, SSD1306 OLED), or a 4.7 kΩ pull-up pair. The model will surface "I²C bus" when these clues are present; it cannot prove which chips share a bus without continuity testing.
  • SPI buses (distinct from the SPI flash chip itself) are identified from MOSI/MISO/SCK/CSn silkscreen on a header, or multiple SPI peripherals visible on the same board. The SPI flash chip itself (W25Q / MX25 / etc.) is always called out separately with its CS/CLK/MOSI/MISO pads for in-circuit dumping.
  • Security-relevant discretes: transistors (NPN / PNP — SOT-23 / SOT-89 / TO-92), MOSFETs (SOT-23 small-signal through DPAK / TO-220 power), crystals, RC reset networks, TVS / Schottky diodes, inductors and main power-rail components are surfaced because each maps to a real attack category — voltage glitching, clock glitching, reset glitching, EMFI, brownout, USB power-cycling. The detection's attack_vectors field is contextualised by component role (e.g. an N-channel MOSFET in the VCC path is flagged as a "voltage glitching target"). Generic bypass caps and LED current-limit resistors are skipped to keep the output focused.
  • Attack vectors each get an inline [EMB3D · web · EDB] strip that searches MITRE EMB3D, the open web, and Exploit-DB.
  • Related research per detection links to EMB3D, Google web/Scholar, Exploit-DB, Hackaday and NVD CVEs — parameterised with the detected part number.
  • Quality detector: if the model returns templated/lazy output, a red banner explains exactly which heuristic fired. Hover any flag for the threshold detail.
  • Confidence floor: the slider in Settings hides anything below the chosen confidence. Default 50%.
  • Cross-reference: detected part numbers matched against chips.yaml via exact, family-root, and fuzzy (Fuse.js). The match-kind label on each chip card tells you which fired.
  • Cost & tokens: Analyze button shows the estimated cost before you click; the run stamp in the header shows actual tokens and cost. Ollama is free.
